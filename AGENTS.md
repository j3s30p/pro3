# AGENTS.md
**트레이드오프:** 이 지침들은 속도보다 신중함 쪽으로 치우쳐 있다. 사소한 작업에는 판단껏 적용할 것.

## 1. 코딩하기 전에 먼저 생각하라
**가정하지 마라. 혼란을 숨기지 마라. 트레이드오프를 드러내라.**

구현하기 전에:
- 가정을 명시적으로 밝혀라. 불확실하면 물어보라.
- 해석이 여러 가지로 갈린다면, 말없이 하나를 고르지 말고 모두 제시하라.
- 더 단순한 방법이 있다면 말하라. 근거가 있다면 반론을 제기하라.
- 불분명한 게 있다면 멈춰라. 무엇이 혼란스러운지 짚어라. 물어보라.

## 2. 단순함이 우선이다
**문제를 푸는 최소한의 코드만. 넘겨짚어 만든 건 아무것도 없게.**
- 요청받은 것 이상의 기능은 금지.
- 일회성 코드에 추상화는 금지.
- 요청하지 않은 "유연성"이나 "설정 가능성"은 금지.
- 일어날 수 없는 시나리오에 대한 예외 처리는 금지.
- 200줄을 썼는데 50줄로 가능하다면, 다시 써라.

스스로 물어보라: "시니어 엔지니어가 이걸 보면 과하게 복잡하다고 할까?" 그렇다면 단순화하라.

## 3. 외과수술적 변경
**손대야 하는 것만 손대라. 네가 어지른 것만 치워라.**

기존 코드를 수정할 때:
- 인접한 코드, 주석, 포매팅을 "개선"하지 마라.
- 망가지지 않은 것을 리팩토링하지 마라.
- 네 방식과 다르더라도 기존 스타일을 따라라.
- 관련 없는 죽은 코드(dead code)를 발견하면, 지우지 말고 언급만 하라.

네 변경으로 고아(orphan)가 생겼다면:
- 네 변경 때문에 쓰이지 않게 된 import/변수/함수는 제거하라.
- 요청받지 않은 한, 기존부터 있던 죽은 코드는 제거하지 마라.

판단 기준: 변경된 모든 줄은 사용자의 요청으로 곧바로 추적될 수 있어야 한다.

## 4. 명확한 코드
**코드는 항상 하는 일이 명확해야한다.**
- 코드를 작성하기 전에 구현 할 로직을 설명하라.
- 함수단위로 docstring 또는 주석을 한글로 적어라
- 코드는 명확하게 요청한 기능만을 담고 있어야 한다.
- 네가 생각하기에 필요해 보인다면 추가 기능을 사용자에게 제시하라.


## 5. 목표 주도 실행
**성공 기준을 정의하라. 검증될 때까지 반복하라.**

작업을 검증 가능한 목표로 바꿔라:
- "검증 추가" → "잘못된 입력에 대한 테스트를 작성한 뒤, 통과시켜라"
- "버그 수정" → "버그를 재현하는 테스트를 작성한 뒤, 통과시켜라"
- "X 리팩토링" → "리팩토링 전후로 테스트가 통과하는지 확인하라"

여러 단계로 이뤄진 작업은 간단한 계획을 밝혀라:
```
1. [단계] → 검증: [확인 방법]
2. [단계] → 검증: [확인 방법]
3. [단계] → 검증: [확인 방법]
```

강력한 성공 기준이 있으면 독립적으로 반복 작업할 수 있다. 약한 기준("일단 되게 해")은 끊임없는 추가 확인을 요구한다.
